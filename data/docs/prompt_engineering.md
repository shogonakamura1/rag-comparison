# プロンプトのベストプラクティス

Claude Opus 4.6、Claude Sonnet 4.5、Claude Haiku 4.5を含むClaudeの最新モデルに対するプロンプトエンジニアリング技術。

---

このガイドでは、Claude Opus 4.6、Claude Sonnet 4.5、Claude Haiku 4.5を含むClaudeの最新モデルに対するプロンプトエンジニアリング技術を提供します。これらのモデルは、以前の世代のClaudeモデルよりも正確な指示追従のためにトレーニングされています。

モデルの機能の概要については、[モデル概要](/docs/ja/about-claude/models/overview)をご覧ください。

## 一般原則

### 指示を明確にする

Claudeは明確で具体的な指示に対してよく応答します。望む出力について具体的に記述することで、結果を向上させることができます。「期待以上」の動作を望む場合は、曖昧なプロンプトからモデルに推測させるのではなく、明示的にリクエストしてください。

**効果が低い:**
```
分析ダッシュボードを作成してください
```

**より効果的:**
```
分析ダッシュボードを作成してください。関連する機能やインタラクションをできるだけ多く含めてください。基本を超えて、フル機能の実装を作成してください。
```

### コンテキストを追加してパフォーマンスを向上させる

指示の背景にあるコンテキストや動機を提供すること、例えばそのような動作がなぜ重要なのかをClaudeに説明することで、Claudeがあなたの目標をよりよく理解し、より的確な応答を提供できるようになります。

**効果が低い:**
```
省略記号は絶対に使わないでください
```

**より効果的:**
```
あなたの応答はテキスト読み上げエンジンによって読み上げられるため、省略記号は絶対に使わないでください。テキスト読み上げエンジンは省略記号の発音方法がわからないためです。
```

Claudeは説明から一般化できるほど賢いです。

### 例と詳細に注意を払う

Claudeは正確な指示追従機能の一部として、詳細や例に細心の注意を払います。例が促進したい動作と一致し、避けたい動作を最小限に抑えるようにしてください。

### 長期的な推論と状態追跡

Claudeの最新モデルは、優れた状態追跡機能を備えた長期的な推論タスクに秀でています。Claudeは、一度にすべてを試みるのではなく、一度にいくつかのことに着実に進歩するインクリメンタルな進捗に焦点を当てることで、長時間のセッション全体にわたって方向性を維持します。

#### コンテキスト認識とマルチウィンドウワークフロー

Claude Opus 4.6とClaude 4.5モデルは[コンテキスト認識](/docs/ja/build-with-claude/context-windows#context-awareness-in-claude-sonnet-45-and-haiku-45)を備えており、モデルが会話全体を通じて残りのコンテキストウィンドウ（つまり「トークン予算」）を追跡できます。これにより、Claudeは作業に使えるスペースがどれだけあるかを理解して、タスクの実行とコンテキストの管理をより効果的に行えます。

**コンテキスト制限の管理:**

コンテキストを圧縮したり、外部ファイルにコンテキストを保存できるエージェントハーネスでClaudeを使用している場合、プロンプトの例:

```
あなたのコンテキストウィンドウは制限に近づくと自動的に圧縮され、中断したところから無期限に作業を続けることができます。したがって、トークン予算の懸念からタスクを早期に停止しないでください。トークン予算の制限に近づいたら、コンテキストウィンドウが更新される前に、現在の進捗と状態をメモリに保存してください。常にできるだけ持続的かつ自律的に行動し、予算の終わりが近づいていてもタスクを完全に完了してください。残りのコンテキストに関係なく、タスクを人為的に早期停止しないでください。
```

[メモリツール](/docs/ja/agents-and-tools/tool-use/memory-tool)は、シームレスなコンテキスト遷移のためにコンテキスト認識と自然に組み合わせることができます。

#### マルチコンテキストウィンドウワークフロー

複数のコンテキストウィンドウにまたがるタスクの場合:

1. **最初のコンテキストウィンドウには異なるプロンプトを使用する**: 最初のコンテキストウィンドウを使用してフレームワークを設定し（テストの作成、セットアップスクリプトの作成）、その後のコンテキストウィンドウでToDoリストを反復します。

2. **モデルに構造化された形式でテストを書かせる**: 作業を開始する前にClaudeにテストを作成させ、構造化された形式（例: `tests.json`）で追跡させます。

3. **QOLツールを設定する**: Claudeにセットアップスクリプト（例: `init.sh`）を作成させて、サーバーの起動、テストスイートの実行、リンターを適切に行えるようにします。

4. **新規開始 vs 圧縮**: コンテキストウィンドウがクリアされた場合、圧縮を使用するのではなく、まったく新しいコンテキストウィンドウから開始することを検討してください。

5. **検証ツールを提供する**: 自律的なタスクの長さが増すにつれて、Claudeは継続的な人間のフィードバックなしに正確性を検証する必要があります。

6. **コンテキストの完全な使用を促す**: コンポーネントを効率的に完了してから次に進むようClaudeに促してください。

#### 状態管理のベストプラクティス

- **状態データには構造化された形式を使用する**: 構造化された情報を追跡する場合、JSONやその他の構造化された形式を使用して、Claudeがスキーマ要件を理解できるようにします
- **進捗メモには非構造化テキストを使用する**: 自由形式の進捗メモは、一般的な進捗とコンテキストの追跡に適しています
- **状態追跡にgitを使用する**: Gitは、何が行われたかのログと復元可能なチェックポイントを提供します。
- **インクリメンタルな進捗を強調する**: Claudeに進捗を追跡し、インクリメンタルな作業に焦点を当てるよう明示的に依頼します

### コミュニケーションスタイル

Claudeの最新モデルは、以前のモデルと比較して、より簡潔で自然なコミュニケーションスタイルを持っています:

- **より直接的で地に足がついている**: 自己称賛的な更新ではなく、事実に基づいた進捗レポートを提供します
- **より会話的**: やや流暢で口語的で、機械的ではありません
- **より簡潔**: プロンプトで指示されない限り、効率のために詳細な要約をスキップすることがあります

## 特定の状況に対するガイダンス

### 冗長性のバランス

Claudeの最新モデルは効率性を重視する傾向があり、ツール呼び出し後の口頭での要約をスキップして、次のアクションに直接進むことがあります。

Claudeに作業中に更新を提供させたい場合:

```
ツール使用を伴うタスクを完了した後、行った作業の簡単な要約を提供してください。
```

### ツール使用パターン

Claudeの最新モデルは正確な指示追従のためにトレーニングされており、特定のツールを使用するための明示的な指示から恩恵を受けます。「いくつかの変更を提案してもらえますか」と言うと、変更を実装することがあなたの意図であったとしても、Claudeは実装するのではなく提案を提供することがあります。

Claudeにアクションを取らせるには、より明示的にしてください:

**効果が低い（Claudeは提案のみ行う）:**
```
この関数を改善するためのいくつかの変更を提案してもらえますか？
```

**より効果的（Claudeが変更を行う）:**
```
この関数のパフォーマンスを改善するように変更してください。
```

デフォルトでClaudeをよりプロアクティブにアクションを取るようにするには、システムプロンプトに以下を追加できます:

```
<default_to_action>
デフォルトでは、変更を提案するだけでなく、実装してください。ユーザーの意図が不明確な場合は、最も有用と思われるアクションを推測して進め、推測する代わりにツールを使用して不足している詳細を発見してください。ツール呼び出し（例: ファイルの編集や読み取り）が意図されているかどうかについてユーザーの意図を推測し、それに応じて行動してください。
</default_to_action>
```

モデルをデフォルトでより慎重にしたい場合:

```
<do_not_act_before_instructions>
変更を行うよう明確に指示されない限り、実装やファイルの変更に飛び込まないでください。ユーザーの意図が曖昧な場合は、アクションを取るのではなく、情報の提供、調査の実施、推奨事項の提供をデフォルトとしてください。ユーザーが明示的にリクエストした場合にのみ、編集、変更、または実装を進めてください。
</do_not_act_before_instructions>
```

### ツールの使用とトリガー

Claude Opus 4.5とClaude Opus 4.6は、以前のモデルよりもシステムプロンプトに対してより敏感です。ツールやスキルのアンダートリガーを減らすためにプロンプトが設計されていた場合、これらのモデルではオーバートリガーする可能性があります。修正方法は、攻撃的な言語を控えめにすることです。

### 自律性と安全性のバランス

ガイダンスがない場合、Claude Opus 4.6はファイルの削除、フォースプッシュ、外部サービスへの投稿など、元に戻すのが難しいアクションや共有システムに影響するアクションを取る可能性があります。Claude Opus 4.6に潜在的にリスクのあるアクションを取る前に確認させたい場合は、プロンプトにガイダンスを追加してください:

```
アクションの可逆性と潜在的な影響を考慮してください。ファイルの編集やテストの実行などのローカルで可逆的なアクションを取ることは推奨されますが、元に戻すのが難しいアクション、共有システムに影響するアクション、または破壊的になり得るアクションについては、進める前にユーザーに確認してください。

確認が必要なアクションの例:
- 破壊的な操作: ファイルやブランチの削除、データベーステーブルのドロップ、rm -rf
- 元に戻すのが難しい操作: git push --force、git reset --hard、公開済みコミットの修正
- 他者に見える操作: コードのプッシュ、PR/issueへのコメント、メッセージの送信、共有インフラストラクチャの変更
```

### 過度な思考と過剰な徹底性

Claude Opus 4.6は、特に高い`effort`設定で、以前のモデルよりも大幅に多くの事前探索を行います。

- **包括的なデフォルトをより的を絞った指示に置き換える。**
- **過剰なプロンプトを削除する。**
- **フォールバックとしてeffortを使用する。**

### 応答のフォーマットを制御する

出力フォーマットの制御に特に効果的な方法がいくつかあります:

1. **何をしないかではなく、何をするかをClaudeに伝える**

2. **XMLフォーマットインジケーターを使用する**

3. **プロンプトスタイルを望む出力に合わせる**

4. **特定のフォーマット設定には詳細なプロンプトを使用する**

マークダウンとフォーマットの使用をより制御するには、明示的なガイダンスを提供してください:

```
<avoid_excessive_markdown_and_bullet_points>
レポート、ドキュメント、技術的な説明、分析、またはその他の長文コンテンツを書く場合は、完全な段落と文を使用して、明確で流れるような散文で書いてください。構成には標準的な段落区切りを使用し、マークダウンは主にインラインコード、コードブロック、およびシンプルな見出しに限定してください。太字や斜体の使用は避けてください。
</avoid_excessive_markdown_and_bullet_points>
```

### 調査と情報収集

Claudeの最新モデルは、優れたエージェント検索機能を示し、複数のソースから情報を効果的に見つけて統合できます。最適な調査結果を得るには:

1. **明確な成功基準を提供する**
2. **ソースの検証を促す**
3. **複雑な調査タスクには、構造化されたアプローチを使用する**

```
この情報を構造化された方法で検索してください。データを収集しながら、いくつかの競合する仮説を立ててください。キャリブレーションを改善するために、進捗メモで信頼度レベルを追跡してください。定期的にアプローチと計画を自己批評してください。情報を永続化し透明性を提供するために、仮説ツリーまたは調査メモファイルを更新してください。この複雑な調査タスクを体系的に分解してください。
```

### サブエージェントのオーケストレーション

Claudeの最新モデルは、ネイティブのサブエージェントオーケストレーション機能が大幅に向上しています。

この動作を活用するには:

1. **明確に定義されたサブエージェントツールを確保する**
2. **Claudeに自然にオーケストレーションさせる**
3. **過剰使用に注意する**: Claude Opus 4.6はサブエージェントに対する強い傾向があり、より単純で直接的なアプローチで十分な状況でもサブエージェントを生成する場合があります。

過剰なサブエージェントの使用が見られる場合は、明示的なガイダンスを追加してください:

```
タスクが並列で実行できる場合、分離されたコンテキストが必要な場合、または状態を共有する必要のない独立したワークストリームを含む場合にサブエージェントを使用してください。単純なタスク、順次操作、単一ファイルの編集、またはステップ間でコンテキストを維持する必要があるタスクの場合は、委任するのではなく直接作業してください。
```

### モデルの自己認識

```
アシスタントはAnthropicによって作成されたClaudeです。現在のモデルはClaude Opus 4.6です。
```

モデル文字列を指定する必要があるLLM搭載アプリの場合:

```
LLMが必要な場合、ユーザーが別途リクエストしない限り、デフォルトでClaude Opus 4.6を使用してください。Claude Opus 4.6の正確なモデル文字列はclaude-opus-4-6です。
```

### 思考の感度

拡張思考が無効な場合、Claude Opus 4.5は「think」という単語とその変形に特に敏感です。「think」を「consider」、「believe」、「evaluate」など、同様の意味を伝える代替語に置き換えることをお勧めします。

### 思考とインターリーブ思考機能を活用する

Claudeの最新モデルは、ツール使用後の振り返りや複雑な多段階推論を含むタスクに特に役立つ思考機能を提供します。

Claude Opus 4.6は[アダプティブ思考](/docs/ja/build-with-claude/adaptive-thinking)（`thinking: {type: "adaptive"}`）を使用し、Claudeがいつ、どの程度思考するかを動的に決定します。Claudeは2つの要因に基づいて思考を調整します: `effort`パラメータとクエリの複雑さです。

```
ツールの結果を受け取った後、その品質を慎重に振り返り、進める前に最適な次のステップを決定してください。思考を使用してこの新しい情報に基づいて計画し反復し、その後最善の次のアクションを取ってください。
```

アダプティブ思考のトリガー動作はプロンプトで制御可能です:

```
拡張思考はレイテンシーを追加するため、回答の品質を意味のある形で改善する場合にのみ使用すべきです。通常、多段階推論を必要とする問題に対してです。迷った場合は、直接応答してください。
```

### ドキュメント作成

Claudeの最新モデルは、印象的なクリエイティブなセンスと強力な指示追従により、プレゼンテーション、アニメーション、ビジュアルドキュメントの作成に秀でています。

```
[トピック]に関するプロフェッショナルなプレゼンテーションを作成してください。思慮深いデザイン要素、視覚的な階層構造、適切な場所に魅力的なアニメーションを含めてください。
```

### 改善されたビジョン機能

Claude Opus 4.5とClaude Opus 4.6は、以前のClaudeモデルと比較してビジョン機能が改善されています。特にコンテキスト内に複数の画像が存在する場合、画像処理とデータ抽出タスクでより優れたパフォーマンスを発揮します。

パフォーマンスをさらに向上させるのに効果的な技術の1つは、Claudeにクロップツールまたは[スキル](/docs/ja/agents-and-tools/agent-skills/overview)を与えることです。

### 並列ツール呼び出しの最適化

Claudeの最新モデルは並列ツール実行に秀でています:

- 調査中に複数の投機的検索を実行します
- コンテキストをより速く構築するために複数のファイルを同時に読み取ります
- bashコマンドを並列で実行します

最大並列効率のためのサンプルプロンプト:

```
<use_parallel_tool_calls>
複数のツールを呼び出す予定で、ツール呼び出し間に依存関係がない場合は、すべての独立したツール呼び出しを並列で行ってください。アクションを順次ではなく同時に実行できる場合は、常にツールを同時に呼び出すことを優先してください。
</use_parallel_tool_calls>
```

### エージェントコーディングでのファイル作成を減らす

Claudeの最新モデルは、特にコードを扱う際に、テストや反復の目的で新しいファイルを作成することがあります。

新規ファイルの作成を最小限に抑えたい場合:

```
反復のために一時的な新しいファイル、スクリプト、またはヘルパーファイルを作成した場合は、タスクの終了時にこれらのファイルを削除してクリーンアップしてください。
```

### 過度な積極性

Claude Opus 4.5とClaude Opus 4.6は、余分なファイルの作成、不要な抽象化の追加、リクエストされていない柔軟性の組み込みなど、過剰なエンジニアリングを行う傾向があります。

過剰エンジニアリングを最小限にするサンプルプロンプト:

```
過剰なエンジニアリングを避けてください。直接リクエストされた変更または明らかに必要な変更のみを行ってください。ソリューションをシンプルで焦点を絞ったものに保ってください:

- スコープ: リクエストされた以上の機能の追加、コードのリファクタリング、「改善」を行わないでください。
- ドキュメント: 変更していないコードにdocstring、コメント、型アノテーションを追加しないでください。
- 防御的コーディング: 発生し得ないシナリオに対するエラーハンドリング、フォールバック、バリデーションを追加しないでください。
- 抽象化: 一回限りの操作のためにヘルパー、ユーティリティ、抽象化を作成しないでください。
```

### フロントエンドデザイン

Claude Opus 4.5とClaude Opus 4.6は、強力なフロントエンドデザインを備えた複雑で実用的なWebアプリケーションの構築に秀でています。ただし、ガイダンスがないと、モデルはユーザーが「AIスロップ」美学と呼ぶ汎用的なパターンにデフォルトする可能性があります。

フロントエンド美学のためのサンプルプロンプト:

```
<frontend_aesthetics>
あなたは汎用的な「分布上の」出力に収束する傾向があります。フロントエンドデザインでは、これがユーザーが「AIスロップ」美学と呼ぶものを生み出します。これを避けてください: 驚きと喜びを与えるクリエイティブで独特なフロントエンドを作成してください。

焦点を当てるべき点:
- タイポグラフィ: 美しく、ユニークで、興味深いフォントを選んでください。
- カラー＆テーマ: 一貫した美学にコミットしてください。
- モーション: エフェクトやマイクロインタラクションにアニメーションを使用してください。
- 背景: ソリッドカラーにデフォルトするのではなく、雰囲気と奥行きを作り出してください。

汎用的なAI生成の美学を避けてください:
- 使い古されたフォントファミリー（Inter、Roboto、Arial、システムフォント）
- 陳腐なカラースキーム（特に白い背景に紫のグラデーション）
- 予測可能なレイアウトとコンポーネントパターン
</frontend_aesthetics>
```

### テストの通過とハードコーディングへの過度な集中を避ける

Claudeは、より一般的なソリューションを犠牲にしてテストを通過させることに過度に集中したり、標準ツールを直接使用する代わりに複雑なリファクタリングのためにヘルパースクリプトなどの回避策を使用したりすることがあります。

```
利用可能な標準ツールを使用して、高品質で汎用的なソリューションを書いてください。タスクをより効率的に達成するためにヘルパースクリプトや回避策を作成しないでください。テストケースだけでなく、すべての有効な入力に対して正しく動作するソリューションを実装してください。値をハードコードしたり、特定のテスト入力に対してのみ動作するソリューションを作成したりしないでください。代わりに、問題を一般的に解決する実際のロジックを実装してください。
```

### エージェントコーディングにおけるハルシネーションの最小化

Claudeの最新モデルはハルシネーションが少なく、コードに基づいたより正確で根拠のあるインテリジェントな回答を提供します。

```
<investigate_before_answering>
開いていないコードについて推測しないでください。ユーザーが特定のファイルを参照している場合、回答する前にそのファイルを必ず読んでください。コードベースに関する質問に回答する前に、関連するファイルを調査して読んでください。正しい回答が確実でない限り、調査する前にコードについていかなる主張もしないでください。根拠のある、ハルシネーションのない回答を提供してください。
</investigate_before_answering>
```

### プリフィル応答からの移行

Claude Opus 4.6以降、最後のアシスタントターンでのプリフィル応答はサポートされなくなりました。モデルの知能と指示追従能力が向上したため、プリフィルのほとんどのユースケースではもはやプリフィルが不要になりました。

一般的なプリフィルのシナリオとその移行方法:

**出力フォーマットの制御** — [構造化出力](/docs/ja/build-with-claude/structured-outputs)機能を使用してください。

**前置きの排除** — 直接的な指示を使用してください: 「前置きなしで直接回答してください。『Here is...』、『Based on...』などのフレーズで始めないでください。」

**不適切な拒否の回避** — Claudeは適切な拒否の判断が大幅に改善されました。

**継続** — 継続をユーザーメッセージに移動し、中断された応答の最後のテキストを含めてください: 「あなたの前回の応答は中断され、`[previous_response]`で終わりました。中断した箇所から続けてください。」

**コンテキストの注入とロールの一貫性** — 非常に長い会話の場合、以前プリフィルされたアシスタントのリマインダーをユーザーターンに注入してください。

### LaTeX出力

Claude Opus 4.6は、数式、方程式、技術的な説明にデフォルトでLaTeXを使用します。プレーンテキストを希望する場合:

```
回答はプレーンテキストのみでフォーマットしてください。LaTeX、MathJax、または\( \)、$、\frac{}{}などのマークアップ記法を使用しないでください。すべての数式は標準的なテキスト文字を使用して記述してください（例: 除算には「/」、乗算には「*」、指数には「^」）。
```

## 移行に関する考慮事項

以前の世代からClaude 4.6モデルに移行する際:

1. **望ましい動作を具体的に指定する**
2. **修飾語を使って指示をフレーミングする**
3. **特定の機能を明示的にリクエストする**
4. **思考設定を更新する**: Claude Opus 4.6は、手動の`budget_tokens`による思考の代わりに[アダプティブシンキング](/docs/ja/build-with-claude/adaptive-thinking)（`thinking: {type: "adaptive"}`）を使用します。
5. **プリフィル応答から移行する**
6. **怠惰防止プロンプトを調整する**: Claude Opus 4.6は大幅にプロアクティブになっており、以前のモデルで必要だった指示に対して過剰に反応する可能性があります。

詳細な移行手順については、[移行ガイド](/docs/ja/about-claude/models/migration-guide)を参照してください。
