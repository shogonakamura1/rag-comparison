# v7: 表認識チャンキング 分析レポート

生成日: 2026-04-10

## 変更内容
**chunker.pyに表認識ロジックを追加。** Markdownの表（`|`で始まる連続行）を検出し、表全体を1つのチャンクとして保持する。表の直前の見出し行も含める。

### 実装の核心
```python
def _extract_blocks(text):
    """テキストを「表ブロック」と「通常テキストブロック」に分解。
    表の連続行を1つの表ブロックにまとめる。"""
```

各チャンクには `metadata["block_type"]` で `"table"` または `"text"` のラベルを付ける。

## 重要な改善: Q6（複数セルの算術）

### v6での失敗
```
RAG: 「料金表では『キャッシュ書き込み』として一種類の料金のみが提示されており、
      期間による区別は明記されていません。」
```
**誤答**: 実際は5分・1時間の2種類が表に存在する。

### v7での成功
```
RAG: 「Claude Sonnet 4.5の1時間キャッシュ書き込み料金は1MTokあたり$6、
      Claude Haiku 4.5の1時間キャッシュ書き込み料金は1MTokあたり$2です。
      したがって、これらのモデルの1時間キャッシュ書き込み料金の差額は1MTokあたり$4です。」
```
**完璧な回答**: 2つのセルから値を取得し、引き算を実行。

### 検証: 表チャンクの取り出し成功
```
search('Sonnet 4.5 1時間キャッシュ書き込み 料金')
→ 1位: block_type=table, source=pricing.md
   "| モデル | 基本入力トークン | 5分キャッシュ書き込み | 1時間キャッシュ書き込み | ..."
   料金表全体（11モデル × 5列）が1チャンクで取得された
```

**v7の改善が機能した証拠** — ヘッダーと全データ行が同じチャンクに含まれているため、列の対応が完全に保たれた。

## 残る失敗: Q2（カウント）

### 状況
v7でも依然として「情報が含まれていません」と回答。

### 原因: 検索バイアス
クエリ「最新モデル比較表で『適応思考: いいえ』のモデルは何個？」に対する検索結果（top 8）：

| 順位 | block_type | source | 距離 |
|:---:|---|---|---|
| 1-7 | text | extended-thinking, prompt-engineering | 0.32-0.36 |
| 8 | table | extended-thinking | 0.366 |
| **圏外** | **table** | **models_overview.md** | **>0.366** |

**問題**: 「適応思考」というキーワードが extended-thinking.md の説明文に多数出現するため、説明文チャンクが上位を占める。models_overview.mdの表チャンクは表全体に「適応思考」が1行しかないため、相対的な類似度が下がる。

### v7の限界
- 表チャンキングは「表が取得できれば」機能する
- しかし「表が取得できない」検索バイアス問題は別問題
- → v8で取り組む必要がある

## 次のステップ: v8案

Q2を解くために必要なのは「**表チャンクを優先的に取得する仕組み**」。

### 候補1: メタデータフィルタリング
質問が「表」「比較表」「何個」「いくつ」などのキーワードを含む場合、`block_type=table` のチャンクを優先的に取得する。

### 候補2: ハイブリッド検索
セマンティック検索 + キーワード完全一致検索を組み合わせる。「適応思考」「いいえ」を含む表チャンクを確実にヒットさせる。

### 候補3: BM25とのアンサンブル
TF-IDFベースのBM25検索を追加し、ベクトル検索の結果と組み合わせる。

### 候補4: 質問書き換え（Query Rewriting）
LLMで質問を書き換え、検索向けの表現にする。「適応思考: いいえ」→「適応思考の対応モデル一覧」など。

最もシンプルで効果的なのは **候補1（メタデータフィルタリング）** または **候補2（ハイブリッド検索）** だろう。
